## ⚙️ 공통 개발 환경 & 도구 전략 (각 레벨에 공통 적용)

### 📌 MCP (Model Context Protocol) 활용 전략

- **역할**: Claude가 외부 도구(DB, 파일시스템, 웹)와 직접 상호작용하도록 확장
- **활용 예시**:
  - `mcp-server-filesystem`: 로컬 PDF/CSV 파일을 Claude가 직접 읽어 분석
  - `@supabase/mcp-server-supabase`: Claude가 Supabase DB를 직접 쿼리 (공식 유지, MySQL MCP 대비 기능 우수) ← **기본 DB MCP**
  - `mcp-server-github`: 코드 리포지터리를 Claude가 읽고 PR 리뷰/코드 분석 자동화
  - Level 3 RAG 프로젝트에서: 벡터 DB MCP 서버 연결로 문서 검색 자동화
- **바이브코딩 연계**: Claude Code에서 MCP 서버를 설정해두면 코드 작성 시 DB 스키마, 파일 내용을 실시간으로 참조 가능 → 개발 속도 대폭 향상
- **DB 선택**: 전 레벨 공통으로 **Supabase(PostgreSQL)** 사용. 이유: 공식 MCP 지원, 무료 500MB, JSONB 지원, Studio 내장, TIMESTAMPTZ로 KST 처리 용이

## 🟢 Level 1. 기초 (Foundation)

**핵심 목표:** 동아리 활동의 페인포인트(Pain-point) 해결, 기초 백엔드(Spring Boot), 외부 API 및 n8n 연동 경험 증명

### 💡 프로젝트 1-1: [동아리 연계] 금융 뉴스 스크랩 및 투자 리서치 자동화 봇

- **배경:** 동아리에서 매일 아침 진행하는 시황/뉴스 스크랩 작업의 비효율성 해결.

- **아키텍처 포인트:**

  - **BE (Spring Boot):** 부원들의 관심 종목 및 섹터 관리 (CRUD).

  - **n8n 파이프라인:** 매일 오전 7시 크롤링 로직 실행.

  - **AI (Perplexity API):** 일반 LLM(OpenAI)의 약점인 '최신 정보 부족'을 해결하기 위해 실시간 검색 특화 AI인 Perplexity API를 연동하여 특정 기업의 간밤 뉴스와 애널리스트 리포트 핵심을 검색 및 요약.

  - **결과:** 요약된 리서치 자료를 동아리 디스코드/카카오톡으로 자동 발송 및 웹 대시보드(Next.js)에 전시.

- **💡 보완 포인트:**
  - **추가 기술**: Redis 캐싱 (동일 뉴스 중복 발송 방지)
  - **MCP 연계**: `mcp-server-filesystem`으로 Claude가 크롤링 결과 파일 직접 분석
  - **GitHub Actions**: n8n 파이프라인 헬스체크 모니터링 추가
  - **Vercel 배포**: 구독 현황 관리 대시보드를 Vercel에 배포
  - **Velog 주제 예시**: "n8n + Perplexity API로 금융 뉴스 자동화 구축기"

### 💡 프로젝트 1-2: [투자 대회 연계] 모의투자 포트폴리오 트래커 & AI 데일리 코멘터리

- **배경:** 동아리 모의투자 대회 시, 참가자들의 수익률을 추적하고 멘토링을 제공.

- **아키텍처 포인트:**

  - **BE (Spring Boot):** 한국투자증권 Open API (또는 Yahoo Finance API)를 연동하여 참가자들의 포트폴리오 실시간 수익률 계산 및 저장.

  - **AI (Claude/OpenAI):** 포트폴리오의 종목 구성 비중과 현재 시장 상황을 텍스트로 넘겨주면, "현재 반도체 비중이 너무 높으니 헷징(Hedging)이 필요함"과 같은 멘토링 코멘터리 생성.

- **💡 보완 포인트:**
  - **추가 기능**: 포트폴리오 스냅샷 히스토리 (일별 수익률 차트)
  - **테스트 전략**: JUnit으로 수익률 계산 로직 단위 테스트 필수
  - **Vercel 배포**: 리더보드 + 수익률 차트 대시보드 Vercel 배포
  - **Velog 주제 예시**: "Spring Boot + 한국투자증권 API 연동 삽질기"

## 🟡 Level 2. 중급 (Intermediate)

**핵심 목표:** MSA(다중 백엔드) 도입, 데이터 분석(Python), 클라우드 스토리지 활용, 시각화/특화 AI 결합

### 💡 프로젝트 2-1: [데이터 분석] 기업 재무제표 시각화 및 퀀트(Quant) 분석기

- **배경:** 기업의 DART 공시 데이터를 가져와 재무 건전성을 분석하고 적정 주가를 도출.

- **아키텍처 포인트:**

  - **BE 1 (Spring Boot):** 사용자 인증, 종목 검색 기록, 관심 기업 DB 관리 (Supabase PostgreSQL 사용).

  - **BE 2 (FastAPI):** Python의 `Pandas`, `NumPy`를 활용해 DART에서 가져온 재무 데이터(자산, 부채, 현금흐름) 분석 및 퀀트 지표(PER, PBR, ROE) 계산.

  - **AI & Cloud:** 분석된 데이터를 바탕으로 Claude가 '재무 건전성 리포트'를 텍스트로 작성. 이를 AWS S3에 PDF 형태로 자동 저장하여 유저에게 다운로드 링크 제공.

- **💡 보완 포인트:**
  - **추가 아키텍처**: Spring Boot ↔ FastAPI 간 통신에 **gRPC 또는 REST** 명확히 선택
  - **AWS 비용 절감**: S3 Presigned URL로 PDF 다운로드 (직접 노출 금지)
  - **보안 강화**: API Gateway + JWT 인증 추가
  - **Velog 주제 예시**: "Spring Boot + FastAPI MSA 연동 with Docker Compose"

### 💡 프로젝트 2-2: 금융 트렌드 소셜 센티먼트(감성) 분석 및 시각화 대시보드

- **배경:** 시장의 탐욕/공포 지수나 특정 종목에 대한 대중의 심리를 파악.

- **아키텍처 포인트:**

  - **n8n 파이프라인:** 트위터, 네이버 종목 토론방 등의 텍스트 데이터를 실시간으로 스크래핑.

  - **BE (FastAPI):** 수집된 텍스트를 자연어 처리(NLP)를 통해 긍정/부정/중립으로 분류(Sentiment Analysis).

  - **특화 AI (Nano Banana 등 이미지 API):** 분석된 키워드와 감성 지수를 바탕으로, 오늘 시장의 분위기를 나타내는 '시각적 워드클라우드'나 '은유적 이미지(예: 폭풍우 치는 바다, 평화로운 들판 등)'를 생성형 AI로 매일 자동 생성하여 대시보드 배경으로 활용.

- **💡 보완 포인트:**
  - **추가 고려**: 트위터 API 유료화 이슈 → Reddit API 또는 네이버 뉴스 API 대안 검토
  - **모니터링**: Grafana + Prometheus로 대시보드 메트릭 시각화 추가
  - **이미지 생성 API**: Nano Banana → **DALL-E 3** 또는 **Stability AI API** 명시 (더 안정적)

## 🔴 Level 3. 고급 (Advanced - 졸업작품/최종 무기)

**핵심 목표:** 금융공기업 실제 업무 적용 가능성, RAG 기술, 대규모 인프라(Docker/CI·CD) 증명

### 💡 프로젝트 3-1: DART 기업 공시 및 금융 규제 컴플라이언스 AI 어시스턴트 (강추)

- **배경:** 금감원 공시나 흩어진 금융 규제 법령을 빠르게 찾아 업무에 적용해야 하는 금융권 현업의 페인포인트 해결.

- **아키텍처 포인트:**

  - **RAG 아키텍처:** 금융 규제 PDF 문서 수십 개를 텍스트 청크로 쪼개어 Vector DB(Pinecone 등)에 저장. 질문 시 가장 관련성 높은 법령을 찾아내어 근거(출처)와 함께 답변.

  - **BE (Spring Boot & FastAPI):** Spring Boot는 사내 시스템을 모사하여 엄격한 권한 관리와 감사 로그(Audit Log)를 남기고, FastAPI는 AI 에이전트 역할을 전담.

  - **클라우드 & 배포:** Docker를 활용해 전체 시스템을 컨테이너화하고, Github Actions를 통해 AWS EC2에 자동 배포 파이프라인 구축.

- **💡 보완 포인트:**
  - **MCP 핵심 활용**: `mcp-server-filesystem` + 커스텀 RAG MCP 서버로 Claude가 직접 법령 문서 검색
  - **CI/CD 고도화**: GitHub Actions → AWS ECS Fargate 자동 배포 (EC2 대신 컨테이너 기반 권장)
  - **보안 강화**: AWS Secrets Manager로 API 키 관리, IAM 역할 기반 접근 제어
  - **취업 연관성 강조**: 금감원/금융결제원 실제 업무 시스템과의 유사성 면접 시 어필 포인트

### 💡 프로젝트 3-2: [자율형 AI] 거시경제 지표 기반 AI 자산 배분 에이전트 (Robo-Advisor)

- **배경:** 단순 코멘터리를 넘어, AI가 주도적으로 시장 데이터를 분석하고 포트폴리오를 리밸런싱.

- **아키텍처 포인트:**

  - **Multi-Agent 시스템:**
    - `Agent A (Perplexity 기반)`: 실시간 글로벌 매크로 지표(미국 CPI, 금리 결정 등) 검색 및 수집.
    - `Agent B (FastAPI 데이터 분석)`: 수집된 지표를 바탕으로 몬테카를로 시뮬레이션 등 금융 공학 알고리즘 실행.
    - `Agent C (Claude 기반)`: 최종 리밸런싱 비중 결정 및 리포트 작성.

  - **n8n 연계:** 이 복잡한 에이전트 간의 워크플로우를 n8n 캔버스 위에서 노드 연결을 통해 시각적으로 구성 및 제어하여 아키텍처의 유연성 확보.

- **💡 보완 포인트:**
  - **MCP + Multi-Agent 연계**: 각 Agent를 MCP 서버로 래핑하여 Claude Orchestrator가 호출
  - **면책 고지**: "투자 권고 아님, 교육/포트폴리오 목적" 명시 (법적 리스크 방지)
  - **백테스팅 추가**: 과거 데이터로 리밸런싱 전략 검증 기능 (신뢰도 향상)

## 📁 실무 폴더 구조 (Professional Folder Structure)

각 Level별 프로젝트마다 폴더 구조가 달라지므로, **실무자들이 범용적으로 사용하는 패턴**을 기준으로 안내.
프로젝트 시작 시 이 구조를 기반으로 필요한 부분만 취사선택.

### 공통 루트 구조 (모든 프로젝트에 적용)

```
project-name/
├── .claude/                     # Claude Code AI 컨텍스트 (토큰 절약)
├── .github/
│   ├── workflows/               # GitHub Actions (CI/CD)
│   ├── PULL_REQUEST_TEMPLATE.md
│   └── ISSUE_TEMPLATE/
├── docs/
│   ├── PRD/                     # 요구사항 정의서 (버전 업데이트 방식)
│   ├── ADR/                     # 아키텍처 결정 기록 (왜 이 기술을 선택했는가)
│   ├── api/                     # API 명세 (서비스별 분리, 버전 업데이트 방식)
│   │   ├── backend/             # Spring Boot REST API 명세
│   │   ├── ai/                  # ai-service (Node.js) API 명세
│   │   └── frontend/            # Frontend API 연동 명세
│   ├── db/
│   │   └── ERD.md               # ERD 전체 문서 (버전 업데이트 방식)
│   └── features/                # 기능 명세서 (역할별 분리)
│       ├── user/                # 사용자 기능 명세 (조회, 등록 등)
│       ├── admin/               # 관리자 기능 명세 (관리자 역할이 있는 경우)
│       └── ai/                  # AI 파이프라인 기능 명세 (자동화, 알림 등)
├── infra/                       # Docker Compose, 배포 스크립트
├── .env.example                 # 환경변수 샘플 (실제 키 제외)
└── README.md
```

### Spring Boot 백엔드 표준 구조 (도메인 주도 패키징)

```
src/main/java/com/example/{project}/
├── config/          # SecurityConfig, WebConfig, SwaggerConfig
├── global/
│   ├── exception/   # GlobalExceptionHandler, CustomException
│   ├── response/    # ApiResponse 공통 응답 래퍼
│   └── util/        # 공통 유틸리티
├── domain/
│   └── {feature}/   # 기능별 패키지 (news, portfolio, user 등)
│       ├── controller/
│       ├── service/
│       ├── repository/
│       ├── entity/
│       └── dto/
│           ├── request/
│           └── response/
├── infra/
│   ├── client/      # 외부 API Feign/WebClient
│   └── scheduler/   # 스케줄러
└── resources/
    ├── application.yml
    ├── application-local.yml
    └── application-prod.yml
```

### Next.js 프론트엔드 표준 구조 (App Router + Feature 기반)

```
src/
├── app/
│   ├── (auth)/          # 인증 Route Group
│   ├── (dashboard)/     # 대시보드 Route Group
│   └── api/             # Route Handlers (BFF)
├── components/
│   ├── ui/              # shadcn/ui 원자 컴포넌트
│   └── features/        # 기능별 복합 컴포넌트
├── hooks/               # 커스텀 React Hook
├── store/               # Zustand 전역 상태
├── types/               # TypeScript 인터페이스/타입
├── lib/                 # API 클라이언트, 유틸
└── constants/           # 상수값

### Node.js (Express) AI 연동 서비스 구조 (MCP/AI 오케스트레이터용)

```
src/

├── routes/              # Express 라우터
├── controllers/         # 요청/응답 처리
├── services/

```

### FastAPI (Python) 데이터 분석 서비스 구조

```

├── api/
│   └── v1/
│       └── routers/     # APIRouter 정의
├── core/
│   ├── config.py        # 환경변수 설정 (pydantic-settings)

## 👥 PM & 풀스택 개발자 관점 (실무 시각)

### PM(Product Manager) 관점에서 갖춰야 할 것들

각 프로젝트마다 개발 전 **PRD 작성 의무화** (이미 `docs/PRD/PRD_Template.md` 존재):
- **문제 정의**: 동아리원의 어떤 페인포인트를 해결하는가?
- **성공 지표(KPI)**: 뉴스 봇이라면 "매일 오전 7시 30분 이전 발송 성공률 95% 이상"
- **우선순위 결정**: MoSCoW 기법 (Must / Should / Could / Won't)
- **로드맵 시각화**: GitHub Projects 칸반 보드 + Milestone 기능 활용
- **사용자 피드백 루프**: 동아리원 설문(구글 폼)으로 기능 만족도 측정 → 다음 스프린트 반영

**면접 어필 포인트**: "단순히 개발만 한 것이 아니라, 문제를 정의하고 지표로 검증한 경험"

### 풀스택 개발자 관점에서 갖춰야 할 것들

- **API Contract 먼저 정의**: OpenAPI(Swagger) yaml을 코드 전에 먼저 작성 → 프론트/백엔드 병렬 개발 가능
- **타입 공유 전략**: 백엔드 DTO 스펙을 Swagger로 export → 프론트에서 `openapi-typescript` 도구로 TypeScript 타입 자동 생성
- **환경 분리**: `application-dev.yml`, `application-prod.yml`로 로컬/운영 환경 명확히 구분, `.env.local` / `.env.production` 대응
- **BFF(Backend For Frontend) 패턴**: Next.js `app/api/` Route Handler를 API Gateway 역할로 활용, 외부 API 키 서버사이드에서만 노출
